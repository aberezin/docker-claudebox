# 3.0 migration guide

**Status:** In progress (2026-07-19). Grows as each 3.0-bundle issue lands.
**Target:** 3.0.0 tag once #1, #5, #10, #11 all land as one coordinated release.
**Tracked:** Issues [#11](https://github.com/aberezin/docker-claudebox/issues/11),
[#1](https://github.com/aberezin/docker-claudebox/issues/1),
[#5](https://github.com/aberezin/docker-claudebox/issues/5),
[#10](https://github.com/aberezin/docker-claudebox/issues/10).

## What 3.0 breaks

3.0 rebrands the fork `claudebox` → `dridock`, unifies the host/container command
surfaces, introduces a general `features:` system replacing `profiles:`, and splits
git-vs-API auth (SSH-for-git, tokens-for-API-only). Every 2.x surface is affected;
a coordinated single migration handles them together.

## Design decisions (finalized 2026-07-19)

Resolved decisions from issue #11's open questions:

### Naming: what renames vs stays

| Surface | 2.x | 3.0 | Rationale |
|---|---|---|---|
| Wrapper binary (install default) | `claudebox` | `dridock` | Brand rename |
| Env var prefix (user-facing) | `CLAUDEBOX_*` | `DRIDOCK_*` | Brand rename; `CLAUDEBOX_*` accepted as alias for one deprecation cycle |
| Env var prefix (framework-dev) | `CLAUDEBOX_HARNESS_*` | `DRIDOCK_HARNESS_*` | Same alias pattern |
| Env vars (external, not-ours) | `GH_TOKEN`, `ANTHROPIC_API_KEY`, `CLAUDE_CODE_OAUTH_TOKEN`, `NEXT_PUBLIC_*`, etc. | unchanged | Not our contracts |
| Image tag | `claudebox:latest` | `dridock:latest` | Brand rename |
| Image label | `org.claudebox.version` | `org.dridock.version` | Brand rename; `checkversion` reads both |
| Project workspace dir | `.claudebox/` | `.dridock/` | Brand rename; wrapper reads both, prefers `.dridock/` |
| Per-project data dir | `~/.config/claudebox/projects/<id>/` | `~/.config/dridock/projects/<id>/` | Brand rename; migrated by `dridock migrate` |
| **Container helper prefix** | **`cb-*`** | **`cb-*` (kept)** | `cb-` is descriptive "container binary," not the brand. Renaming to `dd-*` would be churn without benefit. |
| **Colima profile prefix** | **`cb-<id>`, `cb-infra`, `cb-net`** | **`cb-*` (kept)** | Same reasoning — `cb-` denotes container binary, not brand |
| Container name derivation | `claude-<sanitized-pwd>` | `claude-*` (kept) | Refers to the Claude Code CLI running inside; not our brand |
| Sidecar file naming | `~/.claude/.<container>-*` | unchanged | `.claude/` is Claude Code CLI's dir, not ours |
| Skill dir | `~/.claude/skills/claudebox/` | `~/.claude/skills/dridock/` | The skill IS about our harness; brand rename |
| Framework-dev fingerprint | `wrapper.sh` contains `^CLAUDEBOX_VERSION=` | Contains `^DRIDOCK_VERSION=` OR `^CLAUDEBOX_VERSION=` | Both accepted for one cycle so a renamed-mid-refactor state doesn't lose fingerprint detection |

### Version file / semver constant

- `VERSION` file: unchanged filename, contents bumps to `3.0.0` at the FINAL commit
  of the bundle. During 3.0-dev, stays at `2.24.0` so a partial-3.0 wrapper doesn't
  claim to be 3.0.
- Semver constant inside `wrapper.sh`: renamed `CLAUDEBOX_VERSION` → `DRIDOCK_VERSION`.
  The framework-dev fingerprint checks both patterns for one cycle so a claudebot
  running against a partially-renamed workspace still self-identifies.

### Backward compat window

- One deprecation cycle (all of 3.x line accepts `CLAUDEBOX_*` env, reads
  `.claudebox/` dir, honors old image label).
- Removed in 4.0.
- Not urgent — users are expected to run `dridock migrate` at the 3.0 upgrade point;
  the aliases are for "I forgot to migrate one project" grace.

## Migration script (`dridock migrate`)

New subcommand shipped in 3.0. Three modes:

- `dridock migrate` — migrate the current project's workspace + data dir. Idempotent.
- `dridock migrate --all` — enumerate every `cb-*` VM on the Mac, migrate each
  project's workspace + data dir. Skips already-migrated projects silently.
- **In-place auto-migrate** on first `dridock` invocation in a `.claudebox/`-only
  project. Prints a note; can be disabled with `DRIDOCK_NO_AUTO_MIGRATE=1`.

Each project migration is:

1. `.claudebox/config.yml` → `.dridock/config.yml` (contents unchanged; `profiles:`
   is aliased to `features:` per issue #5).
2. `.claudebox/secrets.env` → `.dridock/secrets.env` (contents unchanged; env-var
   contents are external names like `GH_TOKEN` — not renamed).
3. `.claudebox/BRIEF.md` → `.dridock/BRIEF.md` (if it exists).
4. `.claudebox/config.sample.yml` → `.dridock/config.sample.yml`.
5. `.gitignore` line rewrites (`/.claudebox/config.yml` → `/.dridock/config.yml`, etc.).
6. `~/.config/claudebox/projects/<id>/` → `~/.config/dridock/projects/<id>/`.

Container name (`claude-<sanitized-pwd>`) is unchanged — the derivation stays. Colima
profile name (`cb-<id>`) is unchanged.

Post-migration, the workspace still functions on 2.x wrappers reading the
`.claudebox/` fallback (backward-compat), but the canonical location is `.dridock/`.

## User checklist (fills in as phases land)

Rough user-facing steps to upgrade a 2.x install to 3.0. This grows as each phase
locks its behavior.

1. Update the checkout: `git pull` in the harness fork's local clone.
2. Rebuild the image: `make build` (or `dridock harness sync` — same command,
   renamed).
3. Reinstall the wrapper: `./install.sh` — installs `dridock` binary at
   `~/.local/bin/dridock`. Removes or symlinks the old `claudebox` (TBD in Phase 3).
4. Migrate projects: `dridock migrate --all`.
5. Verify: `dridock checkversion --all` — every cb-* VM shows `3.0.0`.
6. Optional: prune the old `~/.config/claudebox/` after confirming everything works.

The harness fork (this repo) is self-migrating — it goes through the same steps as
any other project. Migration script is developed against other-project scenarios
first, then run against this repo as the acceptance test.
